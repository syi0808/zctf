# zctf 추가 개선 설계서

## serde/JSON 대비 성능 우위 확대를 위한 Rust Buffer Generation / Native Bulk Path 최적화

## 1. 목적

이 문서는 `zctf`가 `serde_json`, N-API object materialization, JSON parse/stringify, 그리고 일부 binary serialization 방식보다 더 큰 성능 우위를 얻기 위한 다음 개선 범위를 정의한다.

현재 `zctf`는 다음 영역에서 이미 성능 우위를 보인다.

```txt
- Rust → JS result return/view
- JSON 대비 first/partial access
- numeric full traversal with bulk API
- string byte length / byte prefix filter
- JS → Rust config single-call sync temp
- JS → Rust config medium/large/plugin-heavy owned path
- config reuse
- mutable JS write → Rust consume pipeline
```

하지만 다음 영역은 여전히 병목이다.

```txt
- Rust result buffer 생성 중 문자열 format/allocation
- 모든 string을 JS string으로 materialize하는 workload
- full toObject materialization
- large buffer 생성의 single-thread fill
- JS DataView loop 기반 aggregate의 한계
```

이 문서의 목표는 `zctf`가 단순히 “빠른 serializer”가 아니라, **serde/JSON이 반드시 수행해야 하는 작업을 생략하거나 더 특화된 방식으로 대체하는 binary view runtime**으로 발전하기 위한 개선 방향을 제시하는 것이다.

---

## 2. 핵심 관점

`zctf`가 `serde`보다 빨라질 수 있는 핵심은 범용 serializer를 더 빠르게 재구현하는 데 있지 않다.

핵심은 다음과 같다.

```txt
serde:
  generic serialize/deserialize
  field traversal
  format-specific encoding
  owned value reconstruction

zctf:
  known schema
  fixed binary layout
  generated offset writer/reader
  lazy access
  partial access
  native bulk operation
  optional materialization
```

즉 `zctf`는 `serde`가 해야 하는 일을 다음 방식으로 줄인다.

```txt
- field name을 쓰지 않는다.
- decimal string conversion을 피한다.
- string literal을 token/id로 대체한다.
- JS object materialization을 피한다.
- Rust owned struct reconstruction을 피한다.
- full traversal은 generated bulk path로 처리한다.
- string workload는 byte-level predicate로 처리한다.
- large buffer는 chunk 단위로 병렬 fill한다.
```

---

## 3. V8 JSON.stringify 최적화에서 가져올 교훈

V8의 최신 `JSON.stringify` 최적화는 `side-effect-free fast path`를 중심으로 한다.

즉, 객체 직렬화 중 사용자 코드 실행, custom `toJSON`, replacer, getter side-effect, GC 등 부작용 가능성이 없다고 판단되면 방어적인 generic path를 피하고 빠른 특화 경로를 사용한다.

`zctf`도 같은 방향을 따른다.

```txt
V8 JSON.stringify fast path:
  side-effect-free object라고 판단되면 generic guard를 줄인다.

zctf fast path:
  plain object / known schema / fixed layout이라고 가정하고
  generic traversal과 dynamic lookup을 제거한다.
```

V8의 문자열 최적화도 참고할 수 있다.

```txt
V8:
  - one-byte / two-byte stringifier 분리
  - SIMD/SWAR로 escape 필요 문자 탐색
  - hidden class 기반 express lane
  - segmented buffer로 큰 문자열 생성 시 realloc/copy 감소

zctf:
  - known ASCII/token path
  - TextEncoder/TextDecoder 회피
  - schema-specific generated compiler
  - chunked/parallel buffer writer
  - native byte predicate
```

따라서 `zctf`의 다음 단계는 다음 문장으로 요약된다.

```txt
Make zctf less like serialization and more like side-effect-free fixed-layout memory generation.
```

## 4. 현재 병목

## 4.1 Rust report generation의 `format!` 비용

현재 report fixture는 package마다 name/version string을 `format!`으로 생성한 뒤 heap에 copy한다.

개념적으로 현재 흐름은 다음과 같다.

```rust
let name = format!("package-{i}");
let version = format!("1.{}.{}", i % 100, i % 10);

write_string(..., &name, ...);
write_string(..., &version, ...);
```

이 방식의 비용:

```txt
- package마다 String allocation
- decimal formatting
- temporary string 생성
- heap에 다시 copy
```

`zctf`의 binary heap은 이미 최종 bytes를 담는 영역이므로, 중간 `String`을 만들 필요가 없다.

---

## 4.2 SoA layout의 pre-pass format 비용

SoA report generation에서는 name heap length를 계산하기 위해 각 name을 `format!`으로 만들 수 있다.

개념적으로 다음과 같은 코드가 병목이다.

```rust
let names_heap_len: usize = (0..count)
    .map(|i| format!("package-{i}").len())
    .sum();
```

이는 실제 bytes를 쓰기 전부터 불필요한 string allocation을 발생시킨다.

---

## 4.3 large buffer generation의 single-thread fill

현재 buffer generation은 대부분 다음 구조다.

```txt
allocate full buffer
cursor = heap_offset

for i in 0..count:
  write string table
  write string heap
  write package record
```

이 방식은 단순하지만, 1,000,000개 이상의 record에서는 하나의 thread가 모든 record/table/heap을 채운다.

큰 report 생성 시간이 전체 `Rust → JS return/view` 시간의 상당 부분을 차지한다면 병렬화 여지가 있다.

---

## 4.4 full string materialization

`zctf`는 string byte length나 prefix filter에서는 강하지만, 모든 string을 JS string으로 materialize하는 경우 여전히 불리하다.

이유:

```txt
- TextDecoder 호출 반복
- JS string allocation 대량 발생
- array/object materialization 비용
```

이 workload는 `zctf`의 기본 lazy view 철학과 맞지 않지만, escape hatch로 최적화할 가치는 있다.

---

## 4.5 JS DataView loop 기반 aggregate의 한계

`sumSizes`, `sumNameByteLengths`, `countNamesWithPrefix` 같은 JS bulk API는 기존 `get(i)` loop보다 빠르지만, 여전히 JS loop와 DataView read 비용이 있다.

더 큰 데이터에서는 Rust native aggregate가 JS loop보다 유리할 수 있다.

---

# 5. 개선안

## 5.1 P0 — `format!` 제거

### 목표

Rust report generation에서 중간 `String` allocation을 제거하고, 최종 heap buffer에 ASCII bytes를 직접 쓴다.

### 기존 방식

```rust
let name = format!("package-{i}");
write_string(out, table_offset, heap_offset, id, &name, &mut cursor);
```

### 개선 방식

```rust
write_name_direct(out, table_offset, heap_offset, id, i, &mut cursor);
```

예시 구현:

```rust
fn write_name_direct(
    out: &mut [u8],
    table_offset: usize,
    heap_offset: usize,
    id: usize,
    index: usize,
    cursor: &mut usize,
) {
    let start = *cursor;
    let relative = start - heap_offset;

    out[*cursor..*cursor + 8].copy_from_slice(b"package-");
    *cursor += 8;

    write_decimal_usize(out, cursor, index);

    let len = *cursor - start;

    write_u32(out, table_offset + id * 8, relative as u32).unwrap();
    write_u32(out, table_offset + id * 8 + 4, len as u32).unwrap();
}
```

version도 동일하게 직접 쓴다.

```rust
fn write_version_direct(
    out: &mut [u8],
    table_offset: usize,
    heap_offset: usize,
    id: usize,
    index: usize,
    cursor: &mut usize,
) {
    let start = *cursor;
    let relative = start - heap_offset;

    out[*cursor..*cursor + 2].copy_from_slice(b"1.");
    *cursor += 2;

    write_decimal_usize(out, cursor, index % 100);

    out[*cursor] = b'.';
    *cursor += 1;

    write_decimal_usize(out, cursor, index % 10);

    let len = *cursor - start;

    write_u32(out, table_offset + id * 8, relative as u32).unwrap();
    write_u32(out, table_offset + id * 8 + 4, len as u32).unwrap();
}
```

`write_decimal_usize`는 우선 `itoa::Buffer`를 사용할 수 있다.

```rust
fn write_decimal_usize(out: &mut [u8], cursor: &mut usize, value: usize) {
    let mut buffer = itoa::Buffer::new();
    let text = buffer.format(value);
    out[*cursor..*cursor + text.len()].copy_from_slice(text.as_bytes());
    *cursor += text.len();
}
```

더 공격적인 최적화는 custom decimal writer를 직접 구현한다.

### 적용 대상

```txt
- make_bench_report_with_capacity
- make_bench_report_direct_string_ref
- make_bench_report_soa
```

### 기대 효과

```txt
- String allocation 제거
- format! 비용 제거
- temporary string copy 제거
- Rust buffer generation 시간 감소
```

### 성공 기준

```txt
1,000,000 packages 기준 zctfReturnAndView가 유의미하게 감소해야 한다.
```

## 5.2 P0 — heap length 계산 수식화

### 목표

buffer size 계산 단계에서 `format!` 기반 length 계산을 제거한다.

### 개선

`package-{i}`의 길이는 다음과 같다.

```txt
len("package-") + digits(i)
```

따라서 전체 name heap length는 다음과 같이 계산할 수 있다.

```rust
fn package_names_heap_len(count: usize) -> usize {
    count * 8 + digit_sum(count)
}
```

version도 수식화할 수 있다.

```txt
version = "1." + digits(i % 100) + "." + digits(i % 10)
```

### 적용 대상

```txt
- exact_heap_len
- make_bench_report_soa
- direct string ref layout
- compact report layout
```

### 기대 효과

```txt
- pre-pass allocation 제거
- format! 제거
- exact capacity 계산 유지
```

## 5.3 P0 — chunked parallel report generation

### 목표

대형 report buffer를 chunk 단위로 병렬 생성한다.

### 핵심 원칙

병렬화에서 가장 중요한 것은 shared heap cursor를 없애는 것이다.

나쁜 방식:

```txt
thread들이 하나의 heap_cursor를 atomic fetch_add로 공유
```

좋은 방식:

```txt
1. chunk별 heap length 계산
2. chunk heap base offset prefix sum
3. 각 chunk가 자기 record/table/heap range만 write
```

### 설계

```rust
struct ChunkPlan {
    start: usize,
    end: usize,
    heap_base: usize,
    heap_len: usize,
}
```

### Pass 1: chunk plan 생성

```rust
fn plan_chunks(count: usize, chunk_size: usize) -> Vec<ChunkPlan> {
    let mut chunks = Vec::new();

    let mut start = 0;
    while start < count {
        let end = (start + chunk_size).min(count);
        let heap_len = heap_len_for_range(start, end);

        chunks.push(ChunkPlan {
            start,
            end,
            heap_base: 0,
            heap_len,
        });

        start = end;
    }

    let mut cursor = 0;
    for chunk in &mut chunks {
        chunk.heap_base = cursor;
        cursor += chunk.heap_len;
    }

    chunks
}
```

### Pass 2: buffer allocate

```rust
let total_heap_len = chunks.last()
    .map(|c| c.heap_base + c.heap_len)
    .unwrap_or(0);

let mut out = vec![0u8; heap_offset + total_heap_len];
```

### Pass 3: chunk fill

각 chunk는 다음 세 영역을 쓴다.

```txt
- package records
- string table entries
- string heap bytes
```

개념적 구현:

```rust
chunks.par_iter().for_each(|chunk| {
    fill_chunk(
        out_ptr,
        table_offset,
        items_offset,
        heap_offset + chunk.heap_base,
        chunk,
    )
});
```

Rust borrow checker 때문에 하나의 `Vec<u8>`를 여러 thread에서 disjoint range로 쓰려면 구현 전략이 필요하다.

### 구현 전략

#### 전략 A: chunk-local buffers

가장 안전한 방식.

```txt
각 thread:
  local records
  local table
  local heap 생성

main thread:
  최종 buffer에 copy
```

장점:

```txt
- safe Rust로 구현 쉬움
- disjoint mutable borrow 문제 없음
```

단점:

```txt
- final copy 비용 있음
- peak memory 증가
```

#### 전략 B: unsafe disjoint write

성능 우선 방식.

```txt
최종 Vec<u8>를 먼저 allocate
각 chunk가 절대 겹치지 않는 range에 직접 write
```

장점:

```txt
- copy 없음
- 가장 빠를 가능성
```

단점:

```txt
- unsafe 필요
- range disjoint 보장을 엄격히 해야 함
```

PoC에서는 먼저 전략 A로 검증하고, 효과가 확인되면 전략 B를 실험한다.

### 적용 대상

```txt
- make_bench_report_compact_parallel
- make_bench_report_direct_string_ref_parallel
- make_bench_report_soa_parallel
```

### Benchmark 항목

```txt
zctfReturnAndView:
  compact sequential
  compact parallel local-buffer
  compact parallel unsafe
  direct sequential
  direct parallel
  soa sequential
  soa parallel
```

### 성공 기준

```txt
1,000,000 packages 이상에서 sequential 대비 유의미한 개선.
작은 count에서는 parallel overhead 때문에 fallback.
```

### 동적 threshold

```rust
if count < 100_000 {
    make_bench_report_compact(count)
} else {
    make_bench_report_compact_parallel(count)
}
```

## 5.4 P1 — native aggregate API

### 목표

JS DataView loop를 Rust native aggregate로 대체한다.

현재 JS bulk API는 `get(i)` loop보다 빠르지만, 여전히 JS loop 비용이 있다.

### API 후보

```js
native.sumReportSizes(buffer)
native.sumReportDependencyCounts(buffer)
native.sumReportNameByteLengths(buffer)
native.countReportNamePrefix(buffer, "package-9")
```

### Rust 구현

```rust
pub fn sum_report_sizes(bytes: &[u8]) -> Result<u64, &'static str> {
    let report = ReportView::parse(bytes)?;
    let mut sum = 0u64;

    for i in 0..report.len {
        let item = report.items + i * PACKAGE_SIZE;
        sum += read_u32_unchecked(bytes, item + 8) as u64;
    }

    Ok(sum)
}
```

prefix filter는 string table/heap을 JS string으로 decode하지 않고 byte 비교로 처리한다.

```rust
pub fn count_name_prefix(bytes: &[u8], prefix: &[u8]) -> Result<u32, &'static str> {
    let report = ReportView::parse(bytes)?;
    let mut count = 0;

    for i in 0..report.len {
        let name_id = report.name_id(i);
        let name = report.string_bytes(name_id);

        if name.starts_with(prefix) {
            count += 1;
        }
    }

    Ok(count)
}
```

### 기대 효과

```txt
- JS DataView loop 제거
- string byte predicate를 Rust에서 빠르게 처리
- memchr/SIMD 최적화 적용 가능
- result가 count/sum이면 JS로 작은 값만 반환
```

### 주의점

이 API는 lazy view 철학과 다르게 native helper를 추가하는 것이다. 따라서 기본 path가 아니라 **bulk native escape hatch**로 둔다.

```txt
zctf view:
  generic lazy access

native aggregate:
  hot workload escape hatch
```

## 5.5 P1 — SIMD/SWAR byte predicate

### 목표

string byte predicate를 더 빠르게 한다.

대상 작업:

```txt
- ASCII 여부 확인
- prefix 비교
- equals 비교
- includes/search
- delimiter 탐색
```

Rust에서는 `memchr`, `memmem`, SIMD/SWAR 기반 구현을 실험할 수 있다.

### API 후보

```rust
count_name_prefix_simd(bytes, prefix)
count_name_equals_ascii(bytes, value)
count_name_contains_ascii(bytes, needle)
```

### 적용 위치

```txt
- native aggregate API
- config compiler ASCII validation
- string table validation
```

### 기대 효과

```txt
- JS byte loop보다 빠른 prefix/filter
- string decode 없이 조건 검사 가능
```
