# zctf 다음 개선 설계서

## Rust → JS Result Full Traversal 최적화

## 1. 목적

이 문서는 `zctf`의 다음 개선 범위를 정의한다.

현재 `zctf`는 Rust result를 JS object로 materialize하지 않고 binary buffer + lazy view로 노출하는 데 성공했다. 기존 benchmark에서 Rust → JS result의 첫 representation 획득은 N-API object 및 JSON 대비 큰 이점을 보였다.

그러나 full traversal workload에서는 field 특성에 따라 성능이 나뉜다.

```txt id="2jnuqx"
강한 영역:
- Rust → JS result 반환
- partial/lazy access
- first access
- numeric field access 일부

약한 영역:
- object-like get(i) 기반 full traversal
- 모든 string을 JS string으로 decode하는 workload
- toObject() full materialization
```

따라서 다음 개선 목표는 다음과 같다.

```txt id="m4u1qm"
zctf의 lazy object API는 유지하되,
full traversal workload에는 layout-aware bulk API를 제공한다.
```

---

## 2. 현재 병목 요약

현재 benchmark는 다음 형태의 루프를 사용한다.

```js id="brw8i7"
let sum = 0
for (let i = 0; i < view.packages.length; i++) {
  sum += view.packages.get(i).size
}
```

문자열도 동일하다.

```js id="mxaehj"
let length = 0
for (let i = 0; i < view.packages.length; i++) {
  length += view.packages.get(i).name.length
}
```

이 경로에는 다음 비용이 반복적으로 발생한다.

```txt id="86h4ud"
- length getter가 매번 DataView에서 len을 읽음
- get(i)가 매번 bounds check 수행
- get(i)가 매번 item offset 계산
- get(i)가 매번 PackageInfoView를 새로 생성
- PackageInfoView getter가 다시 doc.u32 호출
- string getter는 string table lookup + slice validation + TextDecoder 호출
```

현재 `FixedListView.get()`은 매번 `createItem()`을 호출하고, benchmark fixture의 `FixedPackageListView`는 `new PackageInfoView(document, itemOffset)`를 생성한다.

따라서 1,000,000 package full traversal에서는 object allocation과 method dispatch가 과도하게 발생한다.

---

## 3. 개선 방향

개선 방향은 세 가지다.

```txt id="68qjvx"
1. lazy object API와 bulk traversal API를 분리한다.
2. full traversal에서는 view object 생성을 피한다.
3. string workload는 decode/materialize와 metadata-only 접근을 분리한다.
```

즉 사용자는 여전히 다음 API를 사용할 수 있다.

```js id="a2pt5v"
view.packages.get(0).name
```

하지만 full scan에는 다음 API를 권장한다.

```js id="85oyv3"
view.packages.sumSizes()
view.packages.sumNameByteLengths()
view.packages.forEachRaw((offset, index) => {})
```

# 4. 개선안 상세

## 4.1 FixedListView length/capacity cache

### 문제

현재 `FixedListView.length`는 매번 document에서 값을 읽는다.

```js id="es6tld"
get length() {
  return this.document.u32(this.offset)
}
```

full traversal에서 `i < view.packages.length`를 사용하면 루프마다 DataView read가 발생한다.

### 개선

constructor에서 `length`와 `capacity`를 캐시한다.

```js id="zjwe1j"
export class FixedListView {
  constructor(document, offset, stride, createItem) {
    this.document = document
    this.offset = offset
    this.stride = stride
    this.createItem = createItem

    this._length = document.u32(offset)
    this._capacity = document.u32(offset + 4)
    this.itemsOffset = document.u32(offset + 12)

    if (document.u32(offset + 8) !== stride) {
      throw new TypeError("unexpected fixed-list stride")
    }

    if (this._length > this._capacity) {
      throw new RangeError("fixed-list length exceeds capacity")
    }

    document.slice(this.itemsOffset, this._capacity * stride)
  }

  get length() {
    return this._length
  }

  get capacity() {
    return this._capacity
  }
}
```

`commitPush()`에서는 `_length`를 함께 갱신한다.

```js id="cku0dz"
commitPush(writeItem) {
  const length = this._length
  const offset = this.itemOffset(length, true)

  writeItem(offset)

  this._length = length + 1
  this.document.setU32(this.offset, this._length)

  return this.createItem(this.document, offset, length)
}
```

### 기대 효과

```txt id="zid96e"
- loop condition의 DataView read 제거
- push 이후 JS view length 일관성 유지
- full traversal에서 소폭 개선
```

### 리스크

```txt id="yxgbwv"
- 외부에서 raw buffer를 직접 수정해 list length를 바꾸면 cached length가 stale해짐
```

PoC에서는 raw mutation보다 view mutation을 기준으로 하므로 허용한다.

## 4.2 Bulk numeric traversal method

### 문제

현재 numeric full traversal은 `packages.get(i).size`를 사용한다. 이때 매번 `PackageInfoView` 객체가 생성된다.

### 개선

generated fixture view에 bulk method를 추가한다.

```js id="4t68nj"
export class FixedPackageListView extends FixedListView {
  sumSizes() {
    const view = this.document.view
    const len = this._length
    let offset = this.itemsOffset + 8
    let sum = 0

    for (let i = 0; i < len; i++) {
      sum += view.getUint32(offset, true)
      offset += 16
    }

    return sum
  }

  sumDependencyCounts() {
    const view = this.document.view
    const len = this._length
    let offset = this.itemsOffset + 12
    let sum = 0

    for (let i = 0; i < len; i++) {
      sum += view.getUint32(offset, true)
      offset += 16
    }

    return sum
  }
}
```

### Benchmark 추가

```js id="iuwu7y"
row.sumSizes = {
  object: measure(() => {
    let sum = 0
    for (const item of object.packages) sum += item.size
    return sum
  }),
  zctfGetLoop: measure(() => {
    let sum = 0
    for (let i = 0; i < view.packages.length; i++) {
      sum += view.packages.get(i).size
    }
    return sum
  }),
  zctfBulk: measure(() => view.packages.sumSizes()),
}
```

### 기대 효과

```txt id="yjqdag"
- PackageInfoView allocation 제거
- FixedListView.get 호출 제거
- itemOffset bounds check 제거
- getter dispatch 감소
- numeric full traversal 개선
```

## 4.3 Raw iteration API

### 문제

bulk method를 field마다 생성하면 빠르지만, API 수가 늘어난다. 범용 runtime에는 field-specific sum method를 넣기 어렵다.

### 개선

`FixedListView`에 raw offset iteration API를 추가한다.

```js id="qej3ud"
forEachRaw(callback) {
  const len = this._length
  let offset = this.itemsOffset

  for (let index = 0; index < len; index++) {
    callback(offset, index, this.document)
    offset += this.stride
  }
}
```

사용 예시:

```js id="oel1kf"
let sum = 0

view.packages.forEachRaw((offset, index, doc) => {
  sum += doc.view.getUint32(offset + 8, true)
})
```

### 기대 효과

```txt id="0w7mfw"
- PackageInfoView allocation 제거
- bounds check 감소
- field-specific generated method보다 범용성 높음
```

### 리스크

callback 호출 비용이 크기 때문에 가장 빠른 방법은 아니다. full traversal hot path에서는 generated bulk method가 더 좋다.

따라서 API 계층을 다음처럼 나눈다.

```txt id="khtbol"
fastest:
  generated bulk method

middle:
  forEachRaw

slowest:
  get(i).field
```

## 4.4 Reusable item view

### 문제

`get(i)`가 매번 새 `PackageInfoView`를 만들기 때문에 allocation이 많다.

### 개선

object-like API를 유지하면서 allocation을 줄이기 위해 cursor/reusable view를 제공한다.

```js id="baslc5"
const cursor = view.packages.cursor()

let sum = 0

for (let i = 0; i < view.packages.length; i++) {
  sum += cursor.moveTo(i).size
}
```

구현:

```js id="bfvev2"
class PackageInfoCursor extends PackageInfoView {
  moveTo(index) {
    this.offset = this.list.itemsOffset + index * this.list.stride
    return this
  }
}
```

`FixedPackageListView`:

```js id="mi5s84"
cursor() {
  return new PackageInfoCursor(this.doc, this.itemsOffset, this)
}
```

### 기대 효과

```txt id="pmfggp"
- PackageInfoView allocation을 1회로 감소
- object-like getter API 유지
```

### 리스크

cursor는 mutable object이므로 보관하면 위험하다.

```js id="w95rmj"
const item = cursor.moveTo(0)
cursor.moveTo(1)
console.log(item.name) // 이제 1번 item을 가리킬 수 있음
```

따라서 API 이름에 cursor semantics를 명확히 드러낸다.

```txt id="lf679i"
cursor는 ephemeral view다.
반복문 안에서만 사용한다.
```

PoC benchmark에는 넣되, public stable API로 노출할지는 별도로 판단한다.

## 4.5 String fast range API

### 문제

현재 string read는 다음 경로를 탄다.

```txt id="zpamto"
doc.string(id)
  → MutableStringTable.get(id)
  → cache 확인
  → range(id)
  → count bounds check
  → slice bounds check
  → TextDecoder.decode
```

full traversal에서는 이 비용이 크다.

### 개선

검증된 document에 대해 unchecked range API를 추가한다.

```js id="xvjq8i"
class MutableStringTable {
  rangeUnchecked(id) {
    const entry = this.tableOffset + id * 8
    const offset = this.heapOffset + this.document.view.getUint32(entry, true)
    const length = this.document.view.getUint32(entry + 4, true)
    return [offset, length]
  }

  getUnchecked(id) {
    const cached = this.document.cache?.get(id)
    if (cached !== undefined) return cached

    const [offset, length] = this.rangeUnchecked(id)
    const value = decoder.decode(this.document.bytes.subarray(offset, offset + length))

    this.document.cache?.set(id, value)

    return value
  }

  byteLengthUnchecked(id) {
    const entry = this.tableOffset + id * 8
    return this.document.view.getUint32(entry + 4, true)
  }
}
```

### 기대 효과

```txt id="zq2fml"
- string id bounds check 제거
- slice validation 제거
- full traversal string metadata 접근 개선
```

### 리스크

invalid buffer에서 out-of-bounds 가능성이 있으므로 일반 `get()`은 유지한다. `getUnchecked()`는 `BinaryDocument` 생성 시 validate가 완료된 document 또는 benchmark fixture 전용으로 사용한다.

## 4.6 String workload benchmark 분리

### 문제

현재 `decodeNames`는 `name.length`를 합산한다.

```js id="gkg094"
for (let i = 0; i < view.packages.length; i++) {
  length += view.packages.get(i).name.length
}
```

이 benchmark는 “모든 name을 JS string으로 materialize”하는 비용과 “length를 알고 싶다”는 목적이 섞여 있다.

### 개선

string workload를 분리한다.

```txt id="wfx1d3"
1. nameByteLengths:
   string table의 byte length만 합산

2. nameDecodeLengths:
   TextDecoder로 JS string을 만든 뒤 length 합산

3. nameMaterializeArray:
   모든 name을 JS string array로 materialize

4. namePrefixFilter:
   bytes 또는 string으로 prefix condition 검사
```

### Benchmark 예시

```js id="hybrvj"
row.names = {
  objectLength: measure(() => {
    let total = 0
    for (const item of object.packages) total += item.name.length
    return total

  zctfGetDecodeLength: measure(() => {
    let total = 0
    for (let i = 0; i < view.packages.length; i++) {
      total += view.packages.get(i).name.length
    }
    return total
  }),

  zctfByteLength: measure(() => view.packages.sumNameByteLengths()),

  zctfMaterializeArray: measure(() => view.packages.materializeNames().length),
}
```

### 기대 효과

```txt id="c3ci6p"
- zctf가 느린 workload와 빠른 workload를 분리해서 해석 가능
- TextDecoder 병목과 string metadata 접근 비용을 따로 측정 가능
```

## 4.7 sumNameByteLengths

### 문제

현재 fixture의 package name은 ASCII 형태다.

```txt id="q0mqwu"
package-0
package-1
package-2
...
```

이 경우 JS string length와 UTF-8 byte length가 같다. 하지만 현재 benchmark는 모든 name을 `TextDecoder`로 decode한다.

### 개선

string table의 length만 읽는 bulk method를 추가한다.

```js id="rkf8y3"
sumNameByteLengths() {
  const doc = this.document
  const view = doc.view
  const strings = doc.strings
  const len = this._length

  let item = this.itemsOffset
  let total = 0

  for (let i = 0; i < len; i++) {
    const id = view.getUint32(item, true)
    total += strings.byteLengthUnchecked(id)
    item += 16
  }

  return total
}
```

### 기대 효과

```txt id="p31v7d"
- TextDecoder 완전 제거
- string table metadata scan 성능 측정 가능
- prefix/filter 계열 최적화의 기반
```

## 4.8 Materialize names batch API

### 문제

모든 string이 실제 JS string으로 필요한 경우, 개별 getter로 접근하면 object/view 생성과 method dispatch가 많다.

### 개선

모든 name을 한 번에 materialize하는 batch method를 추가한다.

```js id="qy9suz"
materializeNames() {
  const result = new Array(this._length)
  const view = this.document.view
  const strings = this.document.strings

  let item = this.itemsOffset

  for (let i = 0; i < result.length; i++) {
    const id = view.getUint32(item, true)
    result[i] = strings.getUnchecked(id)
    item += 16
  }

  return result
}
```

### 기대 효과

```txt id="3vzpcu"
- PackageInfoView allocation 제거
- generic `get(i).name`보다 빠른 full string materialization
- toObject 최적화의 기반
```

## 4.9 toObject 최적화

### 문제

현재 `toObject()`는 내부에서 다시 `packages.get(i).toObject()`를 호출한다.

```js id="1vihe4"
const packages = new Array(this.packages.length)
for (let i = 0; i < packages.length; i++) {
  packages[i] = this.packages.get(i).toObject()
}
```

이 경로는 `PackageInfoView`를 매번 생성한다.

### 개선

`FixedPackageListView.toObjectArray()`를 만들어 직접 record를 읽는다.

```js id="e1z2qs"
toObjectArray() {
  const len = this._length
  const result = new Array(len)
  const view = this.document.view
  const strings = this.document.strings

  let item = this.itemsOffset

  for (let i = 0; i < len; i++) {
    result[i] = {
      name: strings.getUnchecked(view.getUint32(item, true)),
      version: strings.getUnchecked(view.getUint32(item + 4, true)),
      size: view.getUint32(item + 8, true),
      dependencyCount: view.getUint32(item + 12, true),
    }

    item += 16
  }

  return result
}
```

`BenchReportView.toObject()`:

```js id="q8x81g"
toObject() {
  return {
    packageCount: this.packageCount,
    totalSize: Number(this.totalSize),
    durationMs: this.durationMs,
    packages: this.packages.toObjectArray(),
  }
}
```

### 기대 효과

```txt id="z1qjtt"
- toObject full materialization 개선
- PackageInfoView allocation 제거
- string unchecked path 활용 가능
```

## 4.10 Compact readonly buffer mode

### 문제

현재 report buffer는 mutation/list push를 위해 여유 공간을 크게 잡는다.

```txt id="a5i97m"
package capacity = count * 2
string capacity = count * 4
heap capacity = count * 48 + 64
```

read-only traversal benchmark도 이 overallocated mutable buffer를 사용한다. 그 결과 JSON보다 zctf buffer가 더 크다.

### 개선

read-only 전용 compact buffer 생성 함수를 추가한다.

```rust id="axvgmp"
pub fn make_bench_report_compact(count: u32) -> Vec<u8> {
    let package_capacity = count;
    let string_capacity = count * 2;
    let heap_capacity = estimate_exact_initial_heap(count);
    ...
}
```

JS/N-API:

```js id="bwd0n7"
native.makeReportBufferCompact(count)
```

기존 mutable buffer는 유지한다.

```js id="sby4m6"
native.makeReportBufferMutable(count)
```

### 기대 효과

```txt id="fg9afe"
- read-only result buffer size 감소
- cache locality 개선 가능
- full traversal benchmark가 mutable reserve 비용과 분리됨
```

### Benchmark 추가

```js id="ejkbxw"
row.zctfReturnAndView = {
  mutable: measure(() => BenchReportView.from(native.makeReportBuffer(count)).packageCount),
```

## 4.11 Direct StringRef layout 실험

### 문제

현재 record는 string id를 가진다.

```txt id="61mw67"
PackageInfo:
  name_string_id: u32
```

string decode 시 추가 indirection이 필요하다.

```txt id="vkkqiu"
record.name_id
  → string_table[name_id]
  → offset/len
  → heap bytes
```

### 개선 실험

read-only compact layout에서는 record에 string offset/length를 직접 넣어볼 수 있다.

```txt id="kfh7y0"
PackageInfoCompact:
  name_offset: u32
  name_len: u32
  version_offset: u32
  version_len: u32
  size: u32
  dependency_count: u32
```

### 장점

```txt id="dq9m20v"
- string table lookup 제거
- full string traversal 개선 가능
```

### 단점

```txt id="hswvu1"
- record size 증가
- mutable string id model과 다름
- string interning/cache와 충돌 가능
```

따라서 기본 layout 변경이 아니라 benchmark fixture로만 실험한다.

## 4.12 SoA / sidecar column layout 실험

### 문제

현재 layout은 AoS다.

```txt id="cepwj0"
PackageInfo[]
  [name, version, size, dependencyCount]
  [name, version, size, dependencyCount]
```

SoA layout 또는 sidecar column을 추가한다.

```txt id="ozv6hi"
names[]
versions[]
sizes[]
dependencyCounts[]
```

혹은 hybrid:

```txt id="9sj6bz"
AoS package records
+
sidecar sizes[]
+
sidecar dependencyCounts[]
```

### JS API

```js id="lrzgjx"
view.packages.columns.size // Uint32Array or custom column view
```

Benchmark:

```js id="tdi0ys"
let sum = 0
const sizes = view.packages.columns.size
for (let i = 0; i < sizes.length; i++) {
  sum += sizes[i]
}
```

### 기대 효과

```txt id="w3hghz"
- numeric full scan 개선
- TypedArray 기반 scan 가능

```

### 리스크

```txt id="5xyxs2"
- object-like get(i) 접근과 write path 복잡도 증가

```

PoC에서는 별도 layout variant로만 실험한다.

# 5. API 계층 제안

`zctf` result view API는 세 계층으로 나눈다.

## 5.1 Lazy object API

가장 직관적인 API.

```js id="8xwixp"
view.packages.get(i).name
view.packages.get(i).size
```

특징:

```txt id="7vebuc"
- 사용성 좋음
- partial access에 적합
- full traversal에는 상대적으로 느림
```

## 5.2 Cursor/raw API

반복 접근에서 allocation을 줄이는 API.

```js id="j8k56h"
const cursor = view.packages.cursor()

for (let i = 0; i < view.packages.length; i++) {
  sum += cursor.moveTo(i).size
}
```

또는:

```js id="eo9mpb"
view.packages.forEachRaw((offset, index, doc) => {
  sum += doc.view.getUint32(offset + 8, true)
})
```

특징:

```txt id="yqwv6h"
- object allocation 감소
- generated bulk method보다 범용적
- callback/cursor semantics 주의 필요
```

## 5.3 Generated bulk API

가장 빠른 full traversal API.

```js id="r816mr"
view.packages.sumSizes()
view.packages.sumNameByteLengths()
view.packages.materializeNames()
view.packages.toObjectArray()
```

특징:

```txt id="6gdzml"
- 가장 빠름
- schema-specific
- codegen 필요
- full traversal에 적합
```

# 6. Benchmark 변경 계획

# 7. 성공 기준

## 7.1 Numeric traversal

```txt id="b9c4ij"
- zctf bulk sumSizes가 get-loop보다 명확히 빨라야 한다.
- 1,000,000 package 기준 object sumSizes와 같거나 더 빨라지는 것을 목표로 한다.
```

## 7.2 String metadata traversal

```txt id="cjmycn"
- sumNameByteLengths는 기존 get-loop decodeNames보다 큰 폭으로 빨라야 한다.
- TextDecoder를 사용하지 않는 workload에서 zctf의 강점을 보여야 한다.
```

## 7.3 Full string materialization

```txt id="yxtl1g"
- materializeNames/toObjectArray가 기존 get-loop/toObject보다 빨라야 한다.
- object의 이미 materialized string access보다 빠를 필요는 없다.
```

## 7.4 Buffer size

```txt id="4amm9a"
- compact readonly buffer는 기존 mutable buffer보다 작아야 한다.
- return/view benchmark에서 compact mode가 mutable mode보다 같거나 빨라야 한다.
```

# 9. 예상 결과 해석

개선 후에도 다음은 유지된다.

```txt id="x0qtt9"
zctf가 모든 JS object access보다 빠른 것은 아니다.
```

특히 이미 JS object로 materialized 된 string을 반복 접근하는 경우 object가 유리할 수 있다.

하지만 개선 후 목표 메시지는 다음과 같다.

```txt id="gaox0i"
zctf는 partial access뿐만 아니라,
layout-aware bulk traversal에서도 경쟁력을 갖는다.
```

즉 최종 포지셔닝은 다음과 같다.

```txt id="1wwzvg"
partial access:
  lazy object view

full traversal:
  generated bulk API

full materialization:
  optimized toObject / materialize API

metadata/filter:
  string table / byte-level API
```

# 10. 최종 요약

현재 `zctf`의 다음 병목은 JS → Rust config가 아니라 Rust → JS result full traversal이다.

핵심 문제는 다음과 같다.

```txt id="fsjafz"
object-like lazy API를 full scan에도 그대로 사용하고 있다.
```

따라서 다음 개선은 다음 방향으로 진행한다.

```txt id="q34522"
1. FixedListView의 length/capacity를 캐시한다.
2. full traversal 전용 generated bulk API를 추가한다.
3. PackageInfoView allocation 없이 record를 직접 순회한다.
4. string은 decode workload와 metadata workload를 분리한다.
5. unchecked string range/read API를 추가한다.
6. toObject는 get(i).toObject() 대신 direct record read로 최적화한다.
7. readonly compact buffer와 mutable buffer를 분리한다.
8. 이후 SoA/sidecar layout을 실험한다.
```

`zctf`의 다음 개선 방향은 다음 한 문장으로 요약된다.

```txt id="ec6zch"
Keep lazy object views for partial access, but generate layout-aware bulk paths for full traversal.
```
