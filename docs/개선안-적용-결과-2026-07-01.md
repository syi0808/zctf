# zctf 개선안 적용 및 벤치마크 결과

기준 문서: `docs/개선안-2026-07-01-10:14.md`

## 1. 적용 범위

다음 항목을 구현했다.

- report 생성의 `format!`과 중간 `String` allocation 제거
- package name/version의 ASCII byte 직접 쓰기
- name/version heap 길이 수식화
- compact AoS, direct string ref, SoA의 safe local-buffer 병렬 생성
- 100,000건 미만 순차, 이상 병렬인 자동 선택 경로
- Rust native size/dependency/name-byte-length aggregate
- Rust native name prefix byte predicate
- 순차/병렬 결과의 byte-for-byte 동일성 테스트
- JS bulk/JSON/native aggregate 및 순차/병렬 생성 벤치마크

unsafe disjoint write, native full materialization, contains/equals predicate는 이번 범위에 포함하지 않았다.
safe local-buffer 방식만으로 효과를 먼저 검증한다는 설계서의 우선순위를 따랐다.

## 3. Rust → JS compact return/view

`makeReportBufferCompact`와 JS view 생성 전체 시간이다.

| packages | 변경 전 | 변경 후 | 개선 |
|---:|---:|---:|---:|
| 1,000 | 0.144 | 0.035 | 75.7% |
| 10,000 | 0.954 | 0.109 | 88.6% |
| 100,000 | 9.194 | 0.815 | 91.1% |
| 1,000,000 | 93.238 | 5.895 | 93.7% |

1,000,000건 성공 기준인 `zctfReturnAndView`의 유의미한 감소를 충족했다.
직접 byte writer가 대부분의 개선을 만들었고, 100,000건 이상에서는 병렬 생성이 추가 이득을 냈다.

|---:|---|---:|---:|---:|

## 7. 결론과 다음 단계

P0의 `format!` 제거는 1,000,000건 compact return/view를 약 15.8배 빠르게 만들었다.
safe local-buffer 병렬화는 1,000,000건에서 layout별 44.3~51.3%의 추가 개선을 보였다.
native aggregate는 1,000,000건에서 JS bulk 대비 6.68~23.22배 빠르다.

다음 최적화는 우선순위가 명확하다.

1. peak memory와 최종 copy를 줄이는 unsafe disjoint writer를 별도 실험한다.
2. Rust generation-only timing을 N-API return 비용과 분리한다.
3. equals/contains predicate를 추가하고 긴 needle에서 `memchr`/`memmem`을 비교한다.
4. full materialization은 lazy/native aggregate로 대체할 수 없는 실제 workload가 확인될 때 진행한다.
