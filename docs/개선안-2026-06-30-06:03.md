# zctf 개선안 문서

## JS → Rust Config 단일 실행 최적화

## 1. 목적

이 문서는 `zctf` PoC에서 확인된 JS → Rust config 전달 성능을 개선하기 위한 단일 실행 최적화 아이디어를 정리한다.

기존 benchmark 결과에서 `zctf`는 config를 여러 파일에 재사용하는 경우 `compiled buffer`와 `ConfigHandle` 방식으로 매우 큰 성능 이점을 보였다. 반면, single-call 기준에서는 JS config object를 `zctf` binary buffer로 compile하는 비용 때문에 N-API direct object read 또는 JSON 방식보다 빠르지 않았다.

따라서 이 문서의 목표는 다음과 같다.

```txt id="t9cuzk"
목표:
단일 실행에서도 zctf compile+read path의 비용을 줄인다.

핵심 방향:
- generic traversal 제거
- string 처리 비용 제거 또는 축소
- buffer write 수 감소
- Rust read path 단순화
- known schema 기반 specialization 강화
```

이 문서는 config reuse, ConfigHandle, WeakMap cache, content hash cache처럼 여러 호출을 전제로 하는 최적화는 다루지 않는다.

---

## 2. 현재 병목 가정

JS → Rust config single-call path는 대략 다음 구조다.

```txt id="3r2307"
JS config object
  → JS compileConfig(config)
  → zctf binary buffer
  → native.transformCompiled(input, buffer)
  → Rust offset reader
```

현재 single-call에서 비용이 발생하는 위치는 다음과 같다.

```txt id="f9m11n"
1. JS config object traversal
2. optional/default field 확인
3. nested object traversal
4. string literal 처리
5. plugin name / params key string 처리
6. TextEncoder 호출
7. writer buffer allocation/resize
8. presence bitmap / field write
9. Rust generic reader / bounds check / branch
```

단일 실행 최적화는 이 비용을 줄이는 방향으로 진행한다.

---

## 3. 최적화 우선순위 요약

가장 먼저 실험할 항목은 다음 8개다.

```txt id="a51zf3"
P0:
1. Schema-specific generated compiler
2. Fast plain object path
3. String literal → enum id
4. Known plugin/config name id화
5. Default value skip
6. Writer preallocation
7. ASCII fast path + TextEncoder.encodeInto
8. Hot/cold config split + nested scalar flattening
```

# 4. 개선안 상세

## 4.1 Schema-specific generated compiler

### 개요

범용 encoder가 아니라 config schema에 특화된 compiler 함수를 생성한다.

```ts id="w358mw"
compileTransformConfig(config)
```

범용 encoder는 다음과 같은 비용이 있다.

```txt id="1k7bo2"
- schema object 탐색
- field name lookup
- generic type switch
- recursive traversal
- dynamic offset lookup
```

schema-specific compiler는 이 모든 비용을 제거하고, 정해진 offset에 직접 값을 쓴다.

### 예시

```ts id="uzigxk"
export function compileTransformConfig(config: TransformConfig): Uint8Array {
  const writer = new ConfigWriter(estimateTransformConfigSize(config))

  let presence = 0

  if (config.typescript === true) {
    presence |= F_TYPESCRIPT
    writer.u8(OFF_TYPESCRIPT, 1)
  }

  const jsxRuntime = config.jsxRuntime
  if (jsxRuntime !== undefined) {
    presence |= F_JSX_RUNTIME
    writer.u8(OFF_JSX_RUNTIME, encodeJsxRuntime(jsxRuntime))
  }

  writer.u64(OFF_PRESENCE, presence)

  return writer.finish()
}
```

### 기대 효과

```txt id="c8b8jq"
- generic traversal 제거
- dynamic schema lookup 제거
- field별 offset 계산 제거
- V8 inline cache 친화적 property access
- Rust reader와 layout 동기화 쉬움
```

### 우선순위

```txt id="olsfyg"
최상
```

## 4.2 Fast plain object path

### 개요

JS object의 모든 동적 semantics를 지원하지 않고, fast path에서는 plain object만 지원한다.

지원하는 형태:

```txt id="ycg6ch"
- plain object
- own property
- known field
- JSON-like value
- function 없음
- Proxy 없음
- getter side-effect 없음
```

제외하는 형태:

```txt id="mqp2u1"
- Proxy
- class instance
- prototype chain lookup
- getter side-effect 의존
- unknown dynamic field
- symbol key
```

### 예시

느린 방식:

```ts id="f0olfu"
for (const key of Object.keys(config)) {
  switch (key) {
    case "typescript":
      ...
  }
}
```

빠른 방식:

```ts id="kf9ed2"
const typescript = config.typescript
const jsxRuntime = config.jsxRuntime
const plugins = config.plugins
const svgoConfig = config.svgoConfig
```

### 기대 효과

```txt id="k8dlf1"
- Object.keys 제거
- hasOwnProperty 반복 제거
- in operator 제거
- generic dispatch 제거
- direct property access 최적화 가능
```

### 주의점

fast path는 JS object semantics를 완전히 보장하지 않는다. PoC에서는 이를 허용한다.

```txt id="d9cgd3"
Fast path assumes config is a plain object with known fields.
```

### 우선순위

```txt id="h59n7u"
최상
```

## 4.3 Default value skip

### 개요

default 값과 같은 field는 binary buffer에 쓰지 않는다. presence bitmap으로 “명시된 field만” 표시하고, Rust reader가 default를 적용한다.

### 예시

```ts id="m8b0q8"
if (config.typescript === true) {
  presence |= F_TYPESCRIPT
  writer.u8(OFF_TYPESCRIPT, 1)
}
```

`typescript`의 default가 `false`라면 `false`는 쓰지 않는다.

Rust reader:

```rust id="6r7l9c"
fn typescript(&self) -> bool {
    if self.has(F_TYPESCRIPT) {
        self.read_bool(OFF_TYPESCRIPT)
    } else {
        false
    }
}
```

### 기대 효과

```txt id="i31t1a"
- write 수 감소
- buffer size 감소
- default-heavy config에서 compile 비용 감소
- small config에서 특히 유리
```

### 우선순위

```txt id="5upgn7"
높음
```

## 4.4 String literal → enum id

### 개요

config의 string literal union은 문자열로 쓰지 않고 enum id로 변환한다.

대상 예시:

```ts id="u8wjwv"
jsxRuntime: "automatic" | "classic"
exportType: "default" | "named"
```

### 예시

```ts id="gwys46"
const JSX_RUNTIME = {
  automatic: 1,
  classic: 2,
} as const

function encodeJsxRuntime(value: "automatic" | "classic"): number {
  return JSX_RUNTIME[value]
}
```

Binary layout:

```txt id="68puus"
jsx_runtime: u8
```

Rust reader:

```rust id="p59csj"
match self.u8(OFF_JSX_RUNTIME) {
    1 => JsxRuntime::Automatic,
    2 => JsxRuntime::Classic,
    _ => JsxRuntime::Automatic,
}
```

### 기대 효과

```txt id="jipkt2"
- TextEncoder 제거
- string heap write 제거
- Rust string decode 제거
- Rust match/read 단순화
- config hot path 최적화
```


## 4.5 Known plugin/config name id화

### 개요

plugin name, config name, known option name처럼 반복되는 문자열도 id로 변환한다.

대상 예시:

```txt id="yuer33"
"svgo"
"jsx"
"removeViewBox"
"convertColors"
"currentColor"
```

### 예시

```ts id="ir74lg"
const KNOWN_PLUGIN = {
  svgo: 1,
  jsx: 2,
  removeViewBox: 101,
  convertColors: 102,
} as const

function writePluginName(writer: ConfigWriter, name: string) {
  const known = KNOWN_PLUGIN[name as keyof typeof KNOWN_PLUGIN]

  if (known !== undefined) {
    writer.u8(PLUGIN_NAME_KIND, NAME_KIND_KNOWN_ID)
    writer.u16(PLUGIN_NAME_VALUE, known)
    return
  }

  writer.u8(PLUGIN_NAME_KIND, NAME_KIND_STRING_ID)
  writer.u32(PLUGIN_NAME_VALUE, writer.string(name))
}
```

### 기대 효과

```txt id="6v3ljg"
- string-heavy config에서 효과 큼
- known plugin name encoding 제거
- Rust에서 u16 match 가능
- plugin-specific layout 분기 쉬움
```

### 우선순위

```txt id="pxbdfv"
높음
```

## 4.6 ASCII fast path

### 개요

config 문자열은 대부분 ASCII일 가능성이 높다. `TextEncoder`를 호출하기 전에 ASCII 여부를 검사하고, ASCII면 직접 `charCodeAt`으로 byte를 쓴다.

대상 예시:

```txt id="cysj80"
"automatic"
"classic"
"svgo"
"removeViewBox"
"currentColor"
"react"
```

### 예시

```ts id="xgb58a"
function writeAsciiMaybe(
  str: string,
  out: Uint8Array,
  offset: number,
): number {
  for (let i = 0; i < str.length; i++) {
    const c = str.charCodeAt(i)

    if (c > 0x7f) {
      return -1
    }

    out[offset + i] = c
  }

  return str.length
}
```

fallback:

```ts id="uj9slm"
const written = writeAsciiMaybe(value, bytes, heapOffset)

if (written >= 0) {
  heapCursor += written
} else {
  const result = encoder.encodeInto(value, bytes.subarray(heapOffset))
  heapCursor += result.written
}
```

### 기대 효과

```txt id="q2c2b0"
- 짧은 ASCII 문자열에서 TextEncoder 비용 제거
- plugin name/config key/string literal에 유리
- string-heavy config compile 비용 감소
```

### 주의점

ASCII 검사 loop 자체도 비용이다. 따라서 known string은 enum/id화가 우선이고, ASCII fast path는 unknown string fallback에 적용한다.

### 우선순위

```txt id="atk29q"
높음
```

## 4.7 TextEncoder.encodeInto

### 개요

`TextEncoder.encode()`는 새 `Uint8Array`를 생성한다. `encodeInto()`를 사용하면 기존 heap buffer에 직접 UTF-8 bytes를 쓸 수 있다.

느린 방식:

```ts id="3ip6ce"
const encoded = encoder.encode(value)
bytes.set(encoded, heapOffset)
heapCursor += encoded.length
```

개선 방식:

```ts id="6ck6b8"
const dst = bytes.subarray(heapOffset)
const { written } = encoder.encodeInto(value, dst)
heapCursor += written
```

### 기대 효과

```txt id="0ekybi"
- 중간 Uint8Array allocation 감소
- copy 1회 감소 가능
- 긴 문자열 또는 unknown string에 유리
```

### 주의점

`encodeInto()`는 destination capacity가 충분해야 한다. PoC에서는 capacity를 넉넉히 잡고 overflow 시 throw한다.

```txt id="x9tqi8"
PoC rule:
config string heap capacity를 크게 잡는다.
overflow는 throw한다.
```

### 우선순위

```txt id="pakzo9"
중간~높음
```

## 4.8 Writer preallocation

### 개요

config compile 전에 대략적인 buffer 크기를 추정해서 writer buffer를 미리 할당한다. compile 중 buffer resize를 피하는 것이 목표다.

### 예시

```ts id="q89qmw"
function estimateTransformConfigSize(config: TransformConfig): number {
  let size = BASE_CONFIG_SIZE

  if (config.plugins) {
    size += FIXED_LIST_HEADER_SIZE
    size += config.plugins.length * STRING_REF_SIZE
  }

  if (config.svgoConfig?.plugins) {
    size += config.svgoConfig.plugins.length * SVG_PLUGIN_RECORD_SIZE
  }

  size += estimateStringHeapSize(config)

  return align(size, 8)
}
```

사용:

```ts id="ac3wv4"
const writer = new ConfigWriter(estimateTransformConfigSize(config))
```

### 기대 효과

```txt id="tl599s"
- ArrayBuffer resize 감소
- writer growth branch 감소
- copy 비용 감소
- single-call에서도 효과 있음
```

### 우선순위

```txt id="5wftb0"
높음
```

## 4.9 Sync-only temp writer

### 개요

일반 writer pool은 reuse 최적화에 가깝지만, `transform()` 내부에서 compile 후 즉시 native call까지 끝나는 sync-only temp writer는 단일 실행에서도 allocation 감소 효과가 있을 수 있다.

### 예시

```ts id="pdfb1h"
function transform(svg: string, config: TransformConfig) {
  const writer = acquireTempWriter()

  try {
    const bytes = compileTransformConfigInto(writer, config)
    return native.transformCompiled(svg, bytes)
  } finally {
    releaseTempWriter(writer)
  }
}
```

### 조건

```txt id="shqbdp"
- native call이 sync여야 함
- bytes가 native call 이후 사용되지 않아야 함
- async boundary를 넘기면 안 됨
```

### 기대 효과

```txt id="vxu7we"
- writer object allocation 감소
- 내부 buffer 재사용
- hot transform call에서 유리
```

### 우선순위

```txt id="qjx0q6"
조건부 높음
```

## 4.14 Rust schema-specific unchecked reader

### 개요

Rust에서도 generic reader를 제거하고, generated offset reader를 사용한다.

느린 방식:

```rust id="xx37nk"
reader.get_bool("typescript")
reader.get_enum("jsxRuntime")
```

개선 방식:

```rust id="2y6ylo"
#[inline(always)]
fn typescript(&self) -> bool {
    unsafe { self.u8_unchecked(ROOT + OFF_TYPESCRIPT) != 0 }
}

#[inline(always)]
fn jsx_runtime(&self) -> JsxRuntime {
    match unsafe { self.u8_unchecked(ROOT + OFF_JSX_RUNTIME) } {
        1 => JsxRuntime::Automatic,
        2 => JsxRuntime::Classic,
        _ => JsxRuntime::Automatic,
    }
}
```

### 기대 효과

```txt id="5ft93d"
- field name lookup 제거
- bounds check 감소
- branch 감소
- Rust transform hot path 최적화
```

### 우선순위

```txt id="cwwcrp"
높음
```

## 4.15 Rust side local struct 승격

### 개요

단일 실행에서도 transform 내부에서 config를 여러 번 읽는다면, binary view를 계속 offset read하지 않고 Rust local struct로 한 번 승격할 수 있다.

### 예시

```rust id="4fgmg5"
let cfg = TransformConfigLocal::from_zctf(bytes);
transform(svg, &cfg);
```

`TransformConfigLocal`:

```rust id="57149d"
struct TransformConfigLocal {
    typescript: bool,
    jsx_runtime: JsxRuntime,
    export_type: ExportType,
    svgo: bool,
    multipass: bool,
    float_precision: f64,
}
```

### 기대 효과

```txt id="56clp0"
- transform 내부 반복 offset read 제거
- Rust struct field access로 변경
- hot scalar config에 유리
```

### 주의점

config가 작고 transform 내부에서 한두 번만 읽힌다면 오히려 손해일 수 있다.

```txt id="o41t12"
parse cost vs repeated read cost를 벤치로 확인해야 한다.
```

### 우선순위


```txt id="ek7jx1"
1. N-API direct object read
2. JSON.stringify + Rust parse
3. 기존 zctf compile+read
4. 개선 zctf compile+read
```

특히 다음을 분리해서 측정한다.

```txt id="np900y"
- JS compile time
- TextEncoder/ASCII path 비율
- output buffer size
- Rust read time
- transform 전체 시간
```

# 8. 성공 기준

단일 실행 최적화의 성공 기준은 다음과 같다.

```txt id="fmx79i"
1. small/medium config에서 기존 zctf compile+read 대비 명확한 개선
2. large/string-heavy config에서 TextEncoder/string heap 비용 감소
3. Rust read only 0.001ms 수준 유지 또는 개선
4. N-API object/JSON baseline과 single-call에서 경쟁 가능
5. reuse 없이도 zctf path를 선택할 명분 확보
```

현실적인 목표는 다음과 같다.

```txt id="pcrwan"
single-call에서 항상 이기는 것이 아니라,
medium 이상 config에서 baseline과 경쟁 가능하거나,
string-heavy/plugin-heavy config에서 더 빠른 path를 만드는 것.
```

# 9. 최종 요약

단일 실행에서 `zctf` JS → Rust config path를 개선하려면 cache보다 compile path 자체를 줄여야 한다.

핵심 전략은 다음과 같다.

```txt id="qkjx1o"
1. generic traversal 제거
2. schema-specific compiler 사용
3. plain object fast path 제한
4. string literal을 enum id로 대체
5. known plugin/config name을 id화
6. default field write 생략
7. ASCII fast path와 encodeInto로 string write 비용 감소
8. hot/cold split과 nested scalar flattening으로 Rust read path 단순화
9. Rust generated unchecked reader로 bounds/lookup 비용 제거
```

`zctf`의 단일 실행 최적화 방향은 다음 문장으로 요약된다.

```txt id="kzivk1"
Make config compilation less like serialization,
and more like writing a known memory layout.
```
