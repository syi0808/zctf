# JS → Rust 실행 시간 회귀

측정일: 2026-06-30 18:20 KST

이 문서는 `docs/개선안-2026-06-30-06:03.md` 적용 후 정식 release benchmark에서
실행 시간이 증가한 경로만 정리한다. buffer 크기, backing allocation 용량 등 메모리
사용량 변화는 범위에서 제외한다.

근거는 `benchmark-results/napi.json`의 non-quick median 결과다. 단위는 호출당 ms이며,
v1 compiler/reader를 기준으로 v2가 느려진 경우만 포함했다.

## 1. Rust validated read 회귀

| shape | v1 read | v2 validated read | 증가율 | 절대 증가 |
|---|---:|---:|---:|---:|
| small | 0.000035 | 0.000037 | 5.7% | 0.000002 |
| medium | 0.000051 | 0.000062 | 21.6% | 0.000011 |
| large | 0.000131 | 0.000322 | 145.8% | 0.000191 |
| string-heavy | 0.000130 | 0.000264 | 103.1% | 0.000134 |
| plugin-heavy | 0.000131 | 0.000239 | 82.4% | 0.000108 |
| unicode-heavy | 0.000130 | 0.000271 | 108.5% | 0.000141 |
| default-heavy | 0.000032 | 0.000034 | 6.3% | 0.000002 |

관련 개선안은 4.14 `Rust schema-specific unchecked reader`다. 현재 구현은 unchecked
field access 전에 모든 list, name token, string range를 검증한다. 따라서 unchecked
access 자체의 비용이 아니라 v2 reader에 함께 추가된 선행 검증이 회귀 원인이다.
plugin 수가 많은 shape에서 절대·상대 증가가 모두 커지는 결과도 이 설명과 일치한다.

단, 가장 느린 large도 0.000322ms로 기존 성공 기준인 0.001ms 이내다.

## 2. Small config의 owned v2 경로 회귀

| 측정 경로 | v1 | owned v2 | 증가율 | 절대 증가 |
|---|---:|---:|---:|---:|
| compile only | 0.000261 | 0.000322 | 23.4% | 0.000061 |
| compile + Rust read | 0.000306 | 0.000357 | 16.7% | 0.000051 |
| full transform boundary | 0.000375 | 0.000423 | 12.8% | 0.000048 |

이 회귀는 4.1~4.13이 결합된 owned compiler 경로에서 관측됐다. 현재 benchmark만으로
개별 개선안 하나에 원인을 귀속할 수는 없다. small config에는 string/list가 없어
specialized string encoding의 이득이 없고, writer 생성과 v2 compiler의 고정 setup
비용은 그대로 발생한다.

동일 config의 4.9 sync temp 경로는 compile+read 0.000099ms로 v1보다 3.09배
빠르다. 따라서 회귀 범위는 결과 buffer를 매번 소유·할당하는 small path에 한정된다.

## 3. Default-heavy full boundary의 owned v2 회귀

| 측정 경로 | v1 | owned v2 | 증가율 | 절대 증가 |
|---|---:|---:|---:|---:|
| full transform boundary | 0.000458 | 0.000512 | 11.8% | 0.000054 |

default-heavy의 compile only와 compile+read는 각각 v1보다 빨랐지만, input까지 전달하는
full boundary에서만 회귀했다. 절대 차이가 0.000054ms이고 인접 측정과 방향이 다르므로,
특정 개선안의 확정 원인으로 사용하기 전에 반복 실행과 분산 측정이 필요하다.

sync temp full boundary는 0.000202ms로 v1보다 빠르므로 이 회귀도 owned 경로에만
관측됐다.

## 조치 우선순위

1. v2 Rust reader에서 구조 검증과 hot read를 별도 benchmark로 분리한다.
2. 신뢰된 내부 buffer에 검증을 한 번만 수행할 수 있는지 검토한다.
3. small owned compiler에 항목별 on/off ablation benchmark를 추가한다.
4. default-heavy full boundary는 표본 분산과 신뢰구간을 추가한 뒤 회귀 여부를 재판정한다.

현재 결과로 실행 시간 회귀가 관측되지 않은 개선안이나, 메모리 사용량만 변화한 항목은
이 문서에 포함하지 않았다.
